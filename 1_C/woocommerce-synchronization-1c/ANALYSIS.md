# Анализ плагина WooCommerce — обмен с 1С

## Архитектура

### Протокол CommerceML (1С → WordPress)

1. **checkauth** — проверка авторизации (Basic Auth)
2. **init** — инициализация, обмен параметрами
3. **file** — 1С POST-ит содержимое файла в `php://input`; плагин сохраняет во временный каталог
4. **import** — парсинг XML (классификатор, товары, предложения, остатки)
5. **complete** — завершение

**Важно:** 1С **инициирует** обмен и **передаёт** данные на сайт. Наш проект — обратная схема: приложение **запрашивает** данные у 1С по OData.

### Основные файлы

| Файл | Роль |
|------|------|
| `import-1c.php` | Точка входа (в корне WP) |
| `RootProcessStarter.php` | Маршрутизация type/mode, авторизация |
| `CatalogModeFile.php` | Приём файлов от 1С |
| `CatalogModeImport.php` | Запуск парсера XML |
| `ParserXml.php` / `ParserXml31.php` | Парсинг CommerceML |
| `ImagesProduct.php` | Обработка изображений товара |
| `ImageEntity.php` | Вставка/удаление медиа в WP |

## Интересный опыт для нашего каталога

### 1. Обработка изображений

- **Источник:** `element->Картинка` — массив путей (например `import_files/product.jpg`)
- **Дедупликация:** `in_array` по пути, чтобы не дублировать
- **Проверка изменений:** `sha1_file()` + мета `_1c_image_sha_hash` — обновлять только при смене содержимого
- **Галерея:** первый файл — thumbnail, остальные — галерея

**Для нас:** В OData картинки в `Catalog_НоменклатураПрисоединенныеФайлы` — Base64 в XML. Нужно декодировать и кешировать по хешу/ID, а не по пути.

### 2. Прогресс и таймауты

- **HeartBeat:** счётчик обработанных элементов, при превышении лимита — `ProgressException`, ответ `progress`, клиент повторяет запрос
- **Сессия:** `$_SESSION['imagesProgress']` хранит состояние, чтобы продолжить после паузы

**Для нас:** Уже есть пагинация в OData (`$top`, `$skip`). Можно добавить «чекпоинт» по последней обработанной странице.

### 3. Маппинг GUID ↔ сущность

- `_id_1c` — meta продукта/категории, хранит GUID из 1С
- `Product::getSiteProductId($element, $guid)` — поиск по GUID
- Учёт дубликатов: `$_SESSION['IMPORT_1C_PROCESS']['allCurrentProducts']`

**Для нас:** В OData уже есть `Ref_Key` — аналог GUID. Используем как уникальный ключ для кеша и связей.

### 4. Иерархия групп/категорий

- Разбор `Классификатор` → группы
- Рекурсивная структура в XML
- `FindProductCatId` — маппинг группы 1С → WooCommerce-термин

**Для нас:** Группы загружаем по `Parent_Key` (пагинация по родителю). Подход уже реализован.

### 5. Настройки и фильтры

- `SettingsHelper`, хуки `apply_filters` для кастомизации
- `SkipProductByXml` — пропуск товаров без картинки, по атрибутам и т.д.

**Для нас:** Экран настроек (URL, логин, пароль) есть. Можно добавить фильтры (группы, цена мин/макс).

## Что не переносим напрямую

1. **Формат изображений:** CommerceML — отдельные файлы; OData — Base64 в хранилище. Декодирование нужно делать в Flutter.
2. **OData:** Плагин не использует OData, только CommerceML.
3. **Направление обмена:** Плагин принимает данные; наше приложение — получает по запросу.

---

## Резюме

Плагин даёт полезные **концепции** (маппинг GUID, дедупликация изображений по хешу, разбивка на чанки), но не готовый код для OData/Flutter. Основная ценность — понимание типичной логики обмена 1С с внешними системами и приёмы устойчивой обработки больших объёмов.
